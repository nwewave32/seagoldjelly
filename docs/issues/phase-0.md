### P0-001 — 물고기 좌우 반전이 빠르게 깜빡임 (facing flicker)

- **Status:** Verified — 2026-06-28 시뮬레이터에서 깜빡임 해소 확인. 디버그 시각화는 유지(추후 `debug=false`).
- **Severity:** Medium — 크래시는 아니지만 "예쁘게 헤엄친다"(§6 Phase 0 DoD)를 해침
- **Area:** `lib/game/components/fish_component.dart` (render, L56-57) / `lib/game/behaviors/swim_behavior.dart` (update)
- **Spec:** §6 Phase 0 DoD, §1-2 성능/품질
- **보고:** 2026-06-28, 시뮬레이터 확인

**증상**
헤엄 자체는 자연스럽지만, 가끔 물고기의 좌우 방향이 프레임 단위로 빠르게 뒤집힌다(깜빡임). 화면 가장자리 근처에서 더 자주 발생.

**재현**
물고기가 거의 수직(상/하)으로 이동할 때, 특히 가장자리 근처에서 목표 지점에 도달하거나 새 목표를 고르는 순간.

**원인 (근본)**
좌우 반전을 매 프레임 *진행 방향 x성분의 부호* 하나로 결정한다.

```dart
// fish_component.dart
final facingLeft = cos(_facing) < 0;   // _facing = atan2(dir.y, dir.x)
if (facingLeft) canvas.scale(-1, 1);
```

- `dir.x`가 0 근처(거의 수직 이동)일 때, 부호가 0을 사이에 두고 미세하게 떨리며 **매 프레임 facing이 반전**된다. 데드존/히스테리시스가 없다.
- 가장자리 근처는 목표가 margin(24px) 안에서 잡혀 **수직에 가까운 경로**가 자주 생긴다 → 깜빡임 빈발.
- `swim_behavior`의 목표 도달/랜덤 재선정(`wanderiness`) 시 방향이 **즉각 반전**되어 트리거를 더 만든다.
- 즉 "벽 충돌"이 아니라 *수직 이동 + facing 부호 깜빡임*이 진짜 원인.

**제안 수정 (택1 또는 조합)**
1. **데드존 + 히스테리시스(권장, 저비용):** `|dir.x|`가 임계값(예: 0.15) 이상일 때만 facing 갱신. 수직 이동 중엔 직전 facing 유지.
2. **전환 최소 유지시간:** 좌우 전환 후 일정 시간(예: 0.3s)은 재전환 금지.
3. **방향 스무딩:** facing 각도를 목표 방향으로 `turnRate` 기반 보간(lerp)해 급반전 제거. swim_behavior에서 heading을 상태로 들고 부드럽게 회전.

> 1번만으로 대부분 해결될 가능성이 높음. 2/3은 "더 예쁘게"를 위한 추가 폴리시.

**적용한 수정**
제안 1번(데드존 + 히스테리시스) 적용. `fish_component.dart`:
- `_facingLeft` 상태 필드 추가, 매 프레임 부호로 직접 그리지 않음.
- `update()`에서 수평 성분 `|cos(_facing)|`이 `_facingDeadzone(0.15)` 이상일 때만 `_facingLeft` 갱신. 거의 수직 이동 중엔 직전 방향 유지.
- `render()`는 `_facingLeft`만 읽어 반전.

**유력한 2차 원인 (디버그로 검증 중)**
데드존(수직 ±8.6° 콘)은 너무 좁아 대부분의 깜빡임을 못 막는다. 더 근본 원인은
`swim_behavior.update()`의 **목표 오버슈트 진동**으로 추정:
- 목표 12px 이내에 들어오면 새 목표를 `wanderiness*dt*2`(~1.4%/프레임) 확률로만 재선정 → 대부분 프레임은 같은 목표로 계속 기어감.
- 목표에 아주 근접하면 프레임당 이동(~0.6px)이 남은 거리를 넘어서 `toTarget` 부호가 반전 → `atan2` 진행방향이 프레임마다 ~180° 뒤집힘 → 좌우 깜빡임. 위치 무관(중앙에서도) 발생.

**디버그 시각화 추가 (`fish_component.dart`, `debug=true`)**
- 빨간 웨지 = 데드존 콘(좁다는 걸 눈으로 확인용).
- 진행방향 선: 데드존 밖=초록 / 안=노랑.
- 흰 원+선 = 현재 목표. **목표 근처에서 이 선이 급격히 진동하면 오버슈트 진동이 원인 확정.**
- 빨간 잔광 = 좌우 뒤집힌 순간, `flips` = 누적 반전 횟수(빠르게 증가하면 그 지점이 문제).

**다음 후보 수정 (검증 후 택)**
1. 도달 판정 시 **즉시** 새 목표 재선정(확률 제거) + 도달 반경 확대(예: 16~24px).
2. 이동 스텝을 남은 거리로 **클램프**(오버슈트 금지).
3. heading을 상태로 들고 `turnRate`로 **보간**해 급반전 자체를 제거(가장 견고, 권장).
